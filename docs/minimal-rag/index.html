Core

为什么LLM会"胡说八道"？

已经知道：LLM在训练时学到了大量世界知识，能流畅对话。

但是问题是：LLM的训练数据有截止日期，且不包含你的私有文档。当你问它"上周的会议纪要说了什么"，它只能根据统计模式"猜测"，这就是幻觉。

所以这一段要解决：RAG在推理时先从外部文档检索相关信息，注入提示词，让LLM"有据可依"。

纯LLM：不可靠

问："2025年Q3营收？" → 可能编造数字。因为训练数据里没有这份财务报告。

LLM + RAG：有据可依

系统先检索Q3财报片段 → 注入提示词 → LLM基于真实数据回答，并引用来源。

Core

RAG的五步管线

RAG = 检索增强生成。整个流程可以分为五个清晰的步骤：

1. 加载 (Load)

用 TextLoader 读取 PDF、TXT、Markdown 等文档

2. 分块 (Split)

用 RecursiveCharacterTextSplitter 将长文档切成小块（chunk）

3. 嵌入 (Embed)

用 OpenAIEmbeddings 把每个文本块转成向量

4. 存储 & 检索 (Store & Retrieve)

存入 ChromaDB 向量数据库；查询时用相似度搜索找到最相关块

5. 生成 (Generate)

将检索到的文本块注入 RAG Prompt Template，由 LLM 生成最终答案

Hands-on

动手：最小可运行RAG代码

以下代码可以直接复制运行。确保先安装依赖：pip install langchain langchain-openai chromadb

# 1. 加载文档 from langchain_community.document_loaders import TextLoader loader = TextLoader("./my_document.txt") documents = loader.load() # 2. 文本分块 from langchain.text_splitter import RecursiveCharacterTextSplitter splitter = RecursiveCharacterTextSplitter( chunk_size=500, chunk_overlap=50 ) chunks = splitter.split_documents(documents) print(f"Split into {len(chunks)} chunks") # 3. 向量嵌入 from langchain_openai import OpenAIEmbeddings embeddings = OpenAIEmbeddings( model="text-embedding-3-small" ) # 4. 存储到 ChromaDB from langchain_community.vectorstores import Chroma db = Chroma.from_documents( chunks, embeddings, persist_directory="./chroma_db" ) print(f"Stored {db._collection.count()} vectors") # 5. 检索 + 生成 query = "你的问题" docs = db.similarity_search(query, k=3) print(f"Retrieved {len(docs)} chunks") for i, doc in enumerate(docs): print(f"--- Chunk {i+1} ---") print(doc.page_content[:200])

💡 提示：这段代码会生成 ChromaDB 持久化文件到 ./chroma_db/ 目录。实际项目建议设置 OPENAI_API_KEY 环境变量。

Deep Dive

检索质量对比

同样的查询，不同的分块策略和检索参数，结果差异很大：

❌ 糟糕的检索

chunk_size=2000（太大），k=1（太少）
查询："LLM幻觉的原因"
返回：一段关于transformer架构的文字，不相关。

✅ 高质量的检索

chunk_size=500，chunk_overlap=50，k=3
查询："LLM幻觉的原因"
返回：三段关于幻觉原因的文字，高度相关。

🔑 关键参数：

chunk_size：每块最大字符数（300-1000是常见范围）
chunk_overlap：相邻块重叠字符数（避免割裂语义）
k：检索返回的top-k块数量（3-5通常合适）
检索方式：similarity（余弦相似度）vs MMR（最大边际相关性，增加多样性）

Worked Example

完整RAG Chain实现

使用 LangChain 的 chain 把检索和生成串联成一个完整的问答系统：

from langchain.chains import RetrievalQA from langchain_openai import ChatOpenAI from langchain.prompts import PromptTemplate # 自定义 RAG Prompt rag_template = """使用以下上下文片段回答问题。 如果无法从上下文中找到答案，就说"根据提供的文档无法回答"。 上下文： {context} 问题：{question} 回答（请引用具体来源片段）：""" prompt = PromptTemplate( template=rag_template, input_variables=["context", "question"] ) # 创建 LLM llm = ChatOpenAI(model="gpt-4o", temperature=0) # 创建 RetrievalQA Chain qa_chain = RetrievalQA.from_chain_type( llm=llm, chain_type="stuff", retriever=db.as_retriever( search_kwargs={"k": 3} ), chain_type_kwargs={"prompt": prompt} ) # 提问 response = qa_chain.invoke("Q3营收是多少？") print(response["result"])
💡 要点分析： 1. chain_type="stuff" 最简单但受限于上下文长度；大文档可用 "map_reduce"
2. 提示词模板明确要求"无法回答时坦白"——这是防止幻觉的关键技巧
3. temperature=0 降低LLM随机性，让回答更忠实于检索到的文档

Summary

小结：让AI说真话

RAG回答了我们一开始的问题：没有RAG的LLM会胡说八道——因为它只能依赖训练数据中的统计模式。而RAG通过检索真实文档片段，为LLM提供"参考答案"，让生成的内容有据可依。

01 RAG五步法：Load → Split → Embed → Store → Retrieve → Generate。每一步都有关键参数需要根据场景调优。
02 分块是关键：chunk_size决定粒度（300-1000字常见），chunk_overlap避免语义割裂（建议10-20%重叠）。
03 提示词防幻觉：明确告诉LLM"无法从文档回答时就承认"，配合temperature=0，是最有效的幻觉抑制策略。
04 持续迭代：RAG pipeline不是一次搭好就完事的。定期评估检索召回率、生成质量，调整分块策略和检索参数。
