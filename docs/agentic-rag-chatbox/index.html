Chatbox 不是输入框，是任务调度入口

用户只看到“提问→回答”，系统看到的是一条可审计流水线。

User Question→Session Context→Intent Router→Query Rewrite→Retriever→Evidence→Answer Planner→Composer→Verifier→UI
关键判断：如果没有 Intent Router，所有问题都会被同一种检索策略处理，流程类、定义类、查表类问题就会互相污染。

RAG 不是向量库，而是证据供应链

Document Store

存原文和 canonical JSON。

Chunk Store

把文档切成可检索小块。

Retriever

规则、全文、BM25、向量、表格、直接读取。

Evidence Package

citation_id、quote、anchor、source_ref。

Answer Composer

只基于证据生成答案。

关键误区

Vector DB 只是 Retriever 的一种，不是完整 RAG。

Evidence Package：把检索结果变成可引用证据

{
  "citation_id": "c1",
  "document_id": "pep_doc",
  "section_id": "procedure",
  "section_title": "Procedure",
  "quote": "...原文片段...",
  "anchor": "page-12-fragment-3",
  "source_ref": "PEP.pdf#page=12",
  "supports": ["R2 phase review requirement"]
}
为什么重要：没有 Evidence Package，答案就无法验证；有了它，Composer 和 Verifier 才知道每个事实从哪里来。

Answer Planner：先规划答案，再写答案

对于 process_operation，不要直接生成长文。先固定答案骨架，每个 slot 绑定证据。

适用范围

scope evidence

前置条件

input / prerequisite evidence

主流程阶段

procedure evidence

角色责任

role evidence，缺失就标缺证

交付物 / 记录

deliverables / records evidence

review / approval

approval evidence

思考链路展示：不是 raw CoT，是可解释执行轨迹

用户可见：

分析过程
- 问题类型：流程操作
- 检索范围：当前选中文档
- 命中章节：Purpose / Scope / Procedure / Deliverables
- 使用证据：5 条有效引用
- 自检结果：流程证据充足，角色责任证据不足
边界：展示系统做了什么、证据从哪里来、哪里缺证；不展示模型完整内心推理。

你的 Chatbox Framework 模板

1. Input Contract: user_message / session_id / selected_document_ids
2. Intent Router: route / confidence / required_tools / answer_shape
3. Retrieval Plan: query_rewrite / retrievers / rerank_policy
4. Evidence Package: citation_id / quote / anchor / source_ref
5. Answer Planner: slots / evidence_per_slot / missing_evidence
6. Composer: grounded answer only
7. Verifier: claim support / citation validity / hallucination risk
8. UI Renderer: final answer / references / reasoning summary
