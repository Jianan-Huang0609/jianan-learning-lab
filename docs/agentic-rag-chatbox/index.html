从 Chatbox 到 Agentic RAG

用一个 NotebookLM-like 文档问答库项目，讲清楚 Chatbox、意图识别、RAG、工具调用、思考展示与答案校验如何组成可信 QA 系统。

一句话主线

Chatbox 不是输入框，而是 Agentic RAG 的任务入口：它把用户问题转成意图、检索策略、工具调用、证据包、答案计划、引用校验和可解释输出。

适合谁

会写代码但还不理解 RAG / Agent 后端链路的产品、开发、AI 应用同学。

0. 学完你应该能产出什么？

一张 Chatbox 后端架构图能画出从用户问题到答案展示的完整链路。
一份 Intent Route 表能定义 definition、process_operation、evidence_lookup 等问题类型。
一套 Tool Plan 模板知道什么时候调用检索、读上下文、查表、验证引用。
一个 RAG 升级 Checklist知道当前项目下一步该补 Direct Source Read、Eval、Chunk Index、Verifier。

推荐学习路径

  1. 先看 Chatbox 总链路，建立全局地图。
  2. 再学 Intent Recognition，理解用户问题如何被分流。
  3. 接着学 RAG 五层结构，明确“证据供应链”比“向量数据库”更重要。
  4. 然后看 Agentic Design，理解 Router / Planner / Tools / Verifier 如何分工。
  5. 最后用 Workshop 产出自己的 Chatbox 框架草图。

1. Chatbox 背后发生了什么？

用户看到的是“我问一个问题 → AI 回答”，但后端真实链路更像下面这条流水线：

用户问题→会话状态→意图识别→检索策略→工具调用→证据整理→答案规划→生成与校验→前端展示

例子：“PEP 文档的流程如何操作？” 系统不能直接丢给 LLM，而要先判断这是问流程、概念、原文、表格，还是跨文档对比。

2. Intent Recognition：先分流，再执行

definition问“是什么”：PEP 是什么？7.16 是什么？
process_operation问“怎么做/流程/步骤”：PEP 文档的流程如何操作？
evidence_lookup问原文依据：这个要求在哪个章节？
comparison问差异：CT 和 MI 的流程有什么不同？
summary问总结：总结这份文档核心内容。
table_lookup问表格、字段、记录项：有哪些 deliverables？

3. RAG 不是向量数据库

RAG 的核心不是 vector DB，而是：回答之前，先找到可信证据。

Document Store存原始文档和解析后的 canonical document。
Chunk Store把长文档切成适合检索的小块。
Retriever根据问题找相关内容：规则、全文、BM25、向量、表格。
Evidence Package把检索结果变成可引用证据。
Answer Composer只基于证据生成答案。

4. 当前项目的真实链路

当前 Ask Workspace 已经是轻量 RAG，但还不是完整的持久化向量数据库 RAG。

前端调用 /api/session/query
→ 后端读取 Tool/output/parsed/{document_id}.json
→ runtime 构建 section chunks
→ 判断问题 route
→ 改写检索 query
→ HybridRetriever 检索
→ route-aware rerank
→ 构造 evidence package
→ citation validation
→ answer composer 生成回答
→ 返回 evidence_preview / self_check / answer_run
已做文件型 canonical store、runtime chunk、lexical hybrid retrieval、citation validation。
未做persistent chunk index、embedding cache、真实 vector retriever、answer planner、claim verifier。

5. Agentic Design：Agent 不是一个大 Prompt

Agent 更像一个有分工的系统：

Intent Router→Task Planner→Tool Executor→Evidence Builder→Answer Planner→Composer→Claim Verifier→UI Renderer

工具调用本质是后端函数

intent_router(question)
retrieve_sections(question, route="process_operation")
read_context(section_id)
extract_steps(context)
compose_answer(evidence)
verify_claims(answer, evidence)

6. 思考链路怎么展示？

不要展示模型 raw chain-of-thought。应该展示“可解释执行轨迹”。

Reasoning Summary 示例

我识别到这是一个流程操作类问题。检索范围是当前选中文档。命中章节包括 Purpose / Scope / Procedure / Deliverables。共找到 5 条有效引用，其中流程步骤证据充足，但角色责任证据不足。

问题类型检索范围命中章节使用证据自检结果缺证提示

7. 你自己的 Chatbox 框架模板

学完后，可以直接用下面这个模板去设计自己的项目。

Chatbox Framework

1. Input Contract
- user_message
- session_id
- selected_document_ids
- mode: selected_docs / workspace / web / mixed

2. Intent Router
- route
- confidence
- required_tools
- answer_shape

3. Retrieval Plan
- query_rewrite
- retrievers: rule / fulltext / vector / table / direct_read
- rerank_policy

4. Evidence Package
- citation_id
- document_id
- section_id
- quote
- anchor
- source_ref
- support_claims

5. Answer Planner
- answer_slots
- evidence_per_slot
- missing_evidence

6. Composer
- grounded answer only
- citation markers only
- no unsupported claims

7. Verifier
- citation validity
- claim support
- missing constraints
- hallucination risk

8. UI Renderer
- final answer
- references
- reasoning summary
- missing evidence notice

7. Workshop 练习

Lab 1：Intent Classification判断“PEP 是什么？”“R2 阶段 PO 应该做什么？”分别属于哪种 route。
Lab 2：Tool Plan为“R2 阶段 PO 应该做什么？”设计工具调用顺序。
Lab 3：Evidence Package设计 citation_id、document_id、section_id、quote、anchor、source_ref。
Lab 4：Answer Planner为 process_operation 设计答案骨架：范围、前置条件、步骤、角色、交付物、审批、缺证。

8. 项目升级路线图

  1. Direct Source Context Reader：命中 citation 后，回读同 section 前后 fragments。
  2. Retrieval Eval Golden Cases：建立真实问题测试集，避免调参靠感觉。
  3. Persistent Chunk Index：把 runtime chunk 落地成 Tool/output/index/chunks.jsonl。
  4. Embedding Cache / Vector Index：先用本地 embeddings.sqlite 或 embeddings.jsonl。
  5. Answer Planner：先生成答案结构，每个 slot 绑定 evidence。
  6. Claim Verifier：检查每个事实句是否有 citation 支持。
  7. Reasoning Summary UI：展示可解释执行轨迹，而不是 raw CoT。

最终目标：从“能问文档”的轻量 RAG，升级成“可信、可审计、可解释”的 NotebookLM-like Agentic QA。
